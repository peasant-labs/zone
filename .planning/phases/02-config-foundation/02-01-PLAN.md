   IMPORTANT: The spec uses `[harness]` as both top-level sugar (just the string) AND as a TOML table for harness-specific config. The top-level `harness = "claude-code"` is a string sugar. The `[harness]` table contains HarnessConfig fields (version, skip_permissions, etc). BurntSushi/toml cannot have both a string and a table with the same key name. The solution: In the RepoConfig struct, use `Harness string` for the top-level sugar, and use a separate TOML key for the harness section. BUT looking at the spec closely, the top-level `harness = "claude-code"` is sugar for `[zone] harness = "claude-code"`. So a TOML file can have EITHER `harness = "claude-code"` at top level (string) OR `[zone]\nharness = "claude-code"` in the zone section, PLUS `[harness]\nversion = "1.0.26"` as a table. This works because the top-level `harness` key and the `[harness]` table section are different TOML constructs. Actually in TOML, `harness = "claude-code"` at top level and `[harness]` table in the same file is a conflict because `harness` would be both a string and a table. Looking at the spec examples: the minimal config uses `harness = "claude-code"` at top level, while the full config uses `[zone]\nharness = "claude-code"` plus `[harness]\nversion = ""`. So the sugar form and the section form are mutually exclusive in a single file.

   The correct approach: use a raw TOML decode that handles this. Define `HarnessSection` as the `[harness]` table:
   ```go
   type RepoConfig struct {
       Version   int              `toml:"version"`
       Harness   string           `toml:"harness"` // top-level sugar (mutually exclusive with [harness] table for string)
       Zone      ZoneConfig       `toml:"zone"`
       Auth      AuthConfig       `toml:"auth"`
       Workspace WorkspaceConfig  `toml:"workspace"`
       Packages  PackagesConfig   `toml:"packages"`
       Resources ResourcesConfig  `toml:"resources"`
       Network   NetworkConfig    `toml:"network"`
       Hooks     HooksConfig      `toml:"hooks"`
       HarnessCfg HarnessConfig   `toml:"harness"` // THIS WON'T WORK - same toml key
   }
   ```

   Since having `harness` as both a string and a table in the same struct is impossible with the same TOML key, the ACTUAL solution from the spec is: when the user writes `harness = "claude-code"` at top level (sugar), there is NO `[harness]` table. When they use the full form, they write `[zone]\nharness = "claude-code"` plus `[harness]\nversion = ...`. So: define the struct with the `[harness]` table as HarnessConfig. The top-level `harness` string sugar means we need custom handling. Best approach:

   Decode into a struct where the `[harness]` TOML key maps to HarnessConfig (struct). If the user wrote `harness = "claude-code"` (string at top-level), BurntSushi/toml will return a type mismatch error. Catch this error, re-decode without the Harness field, extract the string manually from the raw TOML. Simpler approach: Do a two-pass decode. First pass into a "raw" struct that captures `harness` as `interface{}` to determine if it's a string or table. Then decode properly.

   SIMPLEST approach: Define two config structs. Primary `RepoConfig` has `[harness]` as HarnessConfig. If decode fails with type error on `harness`, try `RepoConfigSugar` which has `Harness string`. Then normalize by copying the string into `Zone.Harness`.

   **LoadGlobal:**
   ```go
   func LoadGlobal() (*GlobalConfig, error) {
       path, err := GlobalConfigPath()
       if err != nil {
           return DefaultGlobalConfig(), nil
       }
       if _, err := os.Stat(path); os.IsNotExist(err) {
           return DefaultGlobalConfig(), nil // missing file = defaults, NOT error
       }
       cfg := &GlobalConfig{}
       md, err := toml.DecodeFile(path, cfg)
       if err != nil {
           return nil, fmt.Errorf("parse global config %s: %w", path, err)
       }
       if undecoded := md.Undecoded(); len(undecoded) > 0 {
           keys := make([]string, len(undecoded))
           for i, k := range undecoded {
               keys[i] = k.String()
           }
           return cfg, &UnknownKeysError{Keys: keys, File: path}
       }
       if cfg.Version == 0 {
           cfg.Version = 1
       }
       return cfg, nil
   }
   ```
