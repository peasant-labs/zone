## User Constraints (from CONTEXT.md)

### Locked Decisions

**Git identity forwarding (DOC-14)**
- Detection: shell out to `git config --global user.name` and `git config --global user.email` via os/exec
- If host has no git config: skip silently — set `.ForwardGitConfig = false`, omit git config lines from entrypoint
- Always-on behavior: no toggle in zone.toml. If host has git identity, forward it; if not, silently omit
- Partial config (only name or only email set): treat as missing — skip forwarding entirely

**Rendered template inspectability**
- Template variable values logged at debug level only (visible with `--debug` or `ZONE_DEBUG=1`)
- Normal output shows nothing about template rendering
- Each rendered file gets a generation header comment: `# Generated by zone <version> -- do not edit`
- Template rendering errors follow spec's actionable error convention (Section 3.12): include the missing field and a hint pointing to zone.toml / global config

**Embed approach**
- Switch from current `embed.FS` to individual string vars per spec Section 11:
  - `var DockerfileTmpl string`
  - `var EntrypointTmpl string`
  - `var ZoneBashrcTmpl string`
- Update `internal/cache/hash.go` (Phase 3 code) to use `templates.DockerfileTmpl` and `templates.EntrypointTmpl` string vars instead of `fs.ReadFile`
- Remove the `embed.FS` export — single access pattern throughout codebase

**Platform detection for macOS (DOC-15)**
- Detection: `runtime.GOOS == "darwin"` + `user.Current().Username` (standard library)
- If detection fails: skip symlink silently (set `.MacOSUsername` to empty string, debug log the failure)
- Username sanitization: spec's `[a-zA-Z0-9_.-]` regex only — no additional length checks
- macOS only: no special handling for Docker Desktop on Linux
- Template data struct gets flat `MacOSUsername string` field — does not embed the full Platform struct (that's for Docker Manager in Phase 5+)

**Container/network naming (DOC-07)**
- Implemented exactly per spec Section 12: `zone-<sanitized-repo-name>-<16-char-sha256-hash>`
- Network name: container name + `-net` suffix
- Container labels: `com.zone.managed`, `com.zone.repo-path`, `com.zone.harness`

**Security (DOC-05, DOC-06)**
- Sudo scoped to exact commands per spec: `/usr/bin/apt-get, /usr/bin/apt, /usr/bin/pip*, /usr/bin/npm, /usr/local/bin/npm`
- Container security flags per spec: `no-new-privileges`, `CapDrop: ALL`, specific CapAdd list, pids limit

**Root/CI mode (DOC-16)**
- When `HostUID == 0`: skip user creation entirely, no USER directive, no sudoers entry
- Template conditional handles this: `{{- if eq .HostUID 0 }}`

## Summary

Phase 4 implements the template rendering engine for zone: populating three Go text/template files (`Dockerfile.tmpl`, `entrypoint.sh.tmpl`, `zone-bashrc.tmpl`) with their full spec content, building a `DockerfileData` struct that aggregates all variables required by those templates, and implementing `ContainerName`/`NetworkName` deterministic naming in `internal/docker/naming.go`. All template content is defined verbatim in spec §11 (lines 947–1148) and §12 (lines 1191–1217).

The embed migration is a mandatory first step: `pkg/templates/templates.go` must drop `embed.FS` and export three `//go:embed` string vars instead. This simultaneously unblocks template rendering and fixes `internal/cache/hash.go`, which currently calls `fs.ReadFile(templates.FS, ...)` — after migration it directly reads `templates.DockerfileTmpl` and `templates.EntrypointTmpl` as strings.

The template data struct (`DockerfileData`) aggregates fields from `config.MergedConfig`, runtime detection (HostUID, MacOSUsername, git identity), and harness-provided values (install commands, health check, entrypoint command). Phase 5 (Harness System) will supply harness-specific fields; Phase 4 must define the struct with placeholder/zero values that compile cleanly, letting Phase 5 fill them in.

**Primary recommendation:** Implement in this order: (1) migrate embed, fix hash.go, (2) fill template files per spec, (3) implement DockerfileData struct + Render* functions, (4) implement naming.go, (5) implement git identity detection, (6) wire rendering to write `.zone/Dockerfile`, `.zone/entrypoint.sh`, `.zone/zone-bashrc`. Tests use assertion-based approach (string Contains checks) rather than golden files, consistent with existing test patterns.

## Standard Stack

### Core

| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| `text/template` | stdlib | Template rendering for Dockerfile and shell scripts | Go stdlib; `html/template` is overkill — no HTML escaping needed for shell/Docker files |
| `embed` | stdlib | Embed template files as string vars at compile time | Zero-runtime-dependency distribution; spec §11 mandates this pattern |
| `os/exec` | stdlib | Shell out to `git config` for identity detection | Standard way to read host git config without importing a git library |
| `os/user` | stdlib | `user.Current()` for HostUID and macOS Username | Zero-dependency; handles UID as string, must convert to int |
| `runtime` | stdlib | `runtime.GOOS` for macOS detection | Standard platform detection |
| `crypto/sha256` + `encoding/hex` | stdlib | ContainerName hash computation | Already used in `internal/cache/hash.go`; consistent pattern |
| `path/filepath` | stdlib | `filepath.Abs`, `filepath.Base` for container naming | Deterministic path normalization |
| `regexp` | stdlib | Sanitize repo name and macOS username to `[a-zA-Z0-9_.-]` | Spec §12 uses this exact regex |
| `fmt` | stdlib | Error wrapping via `fmt.Errorf("context: %w", err)` | Established project convention |
| `strings` | stdlib | Trim, contains, split for output parsing | stdlib |

### Supporting

| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| `github.com/stretchr/testify` | v1.10.0 | Test assertions | All tests — already in go.mod |
| `bytes` | stdlib | `bytes.Buffer` for template rendering to string | Render to buffer, inspect before writing |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| `text/template` | `html/template` | html/template HTML-escapes values; breaks Dockerfile/shell syntax |
| string vars (`//go:embed X var s string`) | `embed.FS` | FS requires `fs.ReadFile` call; string vars are simpler, direct access, match spec exactly |

| assertion-based tests | golden file tests | Golden files need filesystem; assertion-based tests are faster and follow existing project test patterns |

### Pattern 1: Individual String Var Embed

**What:** Each template file is embedded as a named string constant, not via FS.
**When to use:** Always — spec mandates this pattern; simpler than FS.
**Example:**
```go
// pkg/templates/templates.go
package templates

import _ "embed"

//go:embed Dockerfile.tmpl
var DockerfileTmpl string

//go:embed entrypoint.sh.tmpl
var EntrypointTmpl string

//go:embed zone-bashrc.tmpl
var ZoneBashrcTmpl string
```

### Pattern 2: Template Data Struct + Render Function

**What:** All template variables aggregated in a typed struct; `text/template.Execute` writes to a `bytes.Buffer`.
**When to use:** For each of the three templates.
**Example:**
```go
// internal/docker/dockerfile.go
type DockerfileData struct {
    BaseImage             string
    HostUID               int
    AptPackages           []string
    NeedsNode             bool
    NodeVersion           string
    NeedsPython           bool
    PythonVersion         string
    NpmPackages           []string
    PipPackages           []string
    HarnessInstallCommands []string
    HealthCheck           string
    InstallZsh            bool
    Shell                 string
    MacOSUsername         string   // empty string = skip macOS block
    PostInstallCommands   []string
    MountPath             string
}

func RenderDockerfile(data DockerfileData, version string) (string, error) {
    tmpl, err := template.New("Dockerfile").Funcs(templateFuncs()).Parse(templates.DockerfileTmpl)
    if err != nil {
        return "", fmt.Errorf("parse Dockerfile template: %w", err)
    }
    var buf bytes.Buffer
    if err := tmpl.Execute(&buf, data); err != nil {
        return "", fmt.Errorf("render Dockerfile: %w", err)
    }
    return "# Generated by zone " + version + " -- do not edit\n" + buf.String(), nil
}
```

### Pattern 3: Template Funcs for `join`

**What:** The Dockerfile template uses `{{ join .NpmPackages " " }}` — this is NOT a built-in template function. Must be registered via `template.FuncMap`.
**When to use:** Required; template will fail to parse without it.
**Example:**
```go
func templateFuncs() template.FuncMap {
    return template.FuncMap{
        "join": strings.Join,
    }
}
```

### Pattern 4: Deterministic Container Naming

**What:** SHA256 of absolute path → 16-char hex prefix; repo basename sanitized with regex.
**When to use:** `ContainerName()` and `NetworkName()` in `naming.go`.
**Example (verbatim from spec §12):**
```go
func ContainerName(repoPath string) string {
    absPath, _ := filepath.Abs(repoPath)
    hash := sha256.Sum256([]byte(absPath))
    shortHash := hex.EncodeToString(hash[:])[:16]
    repoName := filepath.Base(absPath)
    repoName = regexp.MustCompile(`[^a-zA-Z0-9_.-]`).ReplaceAllString(repoName, "-")
    return fmt.Sprintf("zone-%s-%s", repoName, shortHash)
}

func NetworkName(repoPath string) string {
    return ContainerName(repoPath) + "-net"
}
```

### Pattern 5: Git Identity Detection

**What:** Shell out to `git config --global user.name` / `user.email`; treat partial config as absent.
**When to use:** Called from the function that builds `EntrypointData`.
**Example:**
```go
// DetectGitIdentity returns name, email, and whether to forward.
// Returns (false) if either value is empty.
func DetectGitIdentity() (name, email string, forward bool) {
    name = runGitConfig("user.name")
    email = runGitConfig("user.email")
    if name == "" || email == "" {
        return "", "", false
    }
    return name, email, true
}

func runGitConfig(key string) string {
    out, err := exec.Command("git", "config", "--global", key).Output()
    if err != nil {
        return ""
    }
    return strings.TrimSpace(string(out))
}
```

### Pattern 6: HostUID Detection

**What:** `os/user.Current()` returns `Uid` as a string; parse to int via `strconv.Atoi`.
**When to use:** When building DockerfileData and EntrypointData.
**Example:**
```go
func HostUID() (int, error) {
    u, err := user.Current()
    if err != nil {
        return 0, fmt.Errorf("get current user: %w", err)
    }
    uid, err := strconv.Atoi(u.Uid)
    if err != nil {
        return 0, fmt.Errorf("parse UID %q: %w", u.Uid, err)
    }
    return uid, nil
}
```

### Pattern 7: MacOS Username Detection (DOC-15)

**What:** `runtime.GOOS == "darwin"` gate + `user.Current().Username` + regex sanitize.
**When to use:** Building DockerfileData.MacOSUsername.
**Example:**
```go
func MacOSUsername() string {
    if runtime.GOOS != "darwin" {
        return ""
    }
    u, err := user.Current()
    if err != nil {
        return "" // silently skip on failure
    }
    return regexp.MustCompile(`[^a-zA-Z0-9_.-]`).ReplaceAllString(u.Username, "")
}
```

### Pattern 8: hash.go Migration

**What:** `internal/cache/hash.go` currently calls `fs.ReadFile(templates.FS, "Dockerfile.tmpl")`. After embed migration, replace with direct string var access.
**Critical:** This is a breaking change to an existing passing test (`TestHashChangesOnConfigChange`, etc.). The tests don't check the hash value — they check stability, change detection, and length — so the migration only changes internals, not test expectations.

```go
// After migration:
h.Write([]byte(templates.DockerfileTmpl))
h.Write([]byte(templates.EntrypointTmpl))
// Remove import of "io/fs" — no longer needed
```

### Anti-Patterns to Avoid

- **Using `html/template`:** HTML-escapes `&`, `<`, `>` which breaks shell scripts and Dockerfile RUN commands. Always use `text/template`.
- **Parsing template on every render:** Parse once at package init or function entry, not inside a render loop. For this phase, parse-per-render is fine (called infrequently) but note for clarity.
- **Accessing `templates.FS` after migration:** The `embed.FS` var will be deleted. Any code still referencing `templates.FS` after migration will fail to compile. Check all usages before deleting.
- **`filepath.Abs` ignoring error:** Spec §12 shows `absPath, _ := filepath.Abs(repoPath)`. On Linux/macOS, `filepath.Abs` only errors if `os.Getwd()` fails (extremely rare). This is fine to ignore per spec, but document the decision.
- **Template whitespace gotchas:** The spec templates use `{{-` (trim left whitespace) extensively. Changing this produces extra blank lines in generated output. Copy template content verbatim from spec.
- **Prepending generation header to templates vs appending:** Header comment goes at top of rendered output string, NOT inside the `.tmpl` file itself (otherwise it appears before `# syntax=docker/dockerfile:1` which Docker requires on line 1 of a Dockerfile). Instead, inject header as the first line in the render function output.

**Special case for Dockerfile header:** `# syntax=docker/dockerfile:1` MUST be the first line of the rendered Dockerfile. The generation comment must come AFTER it, not before. Place it on line 2.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Template parsing | Custom string interpolation | `text/template` | Handles conditionals, ranges, whitespace trimming — exactly what spec uses |
| `join` function in templates | Implement from scratch | `strings.Join` in FuncMap | One-liner; Join is already in stdlib |
| File embedding | Read from disk at runtime | `//go:embed` string vars | Distribution-safe; no file path assumptions; compile-time guarantee |
| Platform detection | Parsing `uname` output | `runtime.GOOS` | Reliable, cross-platform, zero overhead |
| UID lookup | Reading `/etc/passwd` | `os/user.Current()` | stdlib; handles all edge cases (LDAP, NIS, etc.) |
| Git identity | Parsing `~/.gitconfig` manually | `git config --global` via exec | Respects includes, conditional includes, env var overrides |

**Key insight:** All problems in this phase have standard library or single-function solutions. The complexity is in assembling correct template data from multiple runtime sources, not in the rendering mechanism itself.

## Common Pitfalls

### Pitfall 1: `join` Missing from Template FuncMap

**What goes wrong:** Template parsing succeeds (the template file isn't loaded until `template.New().Parse()`), but `Execute` panics or returns an error: `function "join" not defined`.
**Why it happens:** `text/template` only knows about built-in functions unless you register extras via `Funcs()`.
**How to avoid:** Always call `.Funcs(templateFuncs())` before `.Parse()`. The FuncMap must be registered before parsing.
**Warning signs:** Error message contains "function 'join' not defined" or similar.

### Pitfall 2: Generation Header Before `# syntax=...` Breaks BuildKit

**What goes wrong:** Docker refuses to use BuildKit syntax, falls back to legacy builder, or errors.
**Why it happens:** The `# syntax=docker/dockerfile:1` parser directive MUST be the first line. Any comment before it (including a generation header) causes Docker to ignore the directive.
**How to avoid:** In `RenderDockerfile`, insert the generation comment as line 2, not line 1. The spec template already starts with `# syntax=docker/dockerfile:1` on line 1 — preserve that ordering.
**Warning signs:** BuildKit not used despite `DOCKER_BUILDKIT=1`, or `docker build` succeeds but `--mount=type=cache` directives fail.

### Pitfall 3: embed Migration Leaves Stale `io/fs` Import in hash.go

**What goes wrong:** `go build` fails with "imported and not used: io/fs".
**Why it happens:** `internal/cache/hash.go` currently imports `io/fs` for `fs.ReadFile`. After replacing with direct string var access, the import is no longer needed.
**How to avoid:** Remove the `io/fs` import and the `templates.FS` reference together in the same change.
**Warning signs:** Compile error referencing unused import.

### Pitfall 4: HostUID=0 Template Conditional Requires Int Comparison

**What goes wrong:** Template reports type mismatch: cannot compare `int` to `0` using `eq` without correct types.
**Why it happens:** Go templates use `eq` for equality but require matching types. If HostUID is stored as `uint32` (from `os/user`) and template uses `{{- if eq .HostUID 0 }}`, this works correctly since `0` is untyped. Using string UID causes comparison failure.
**How to avoid:** Parse UID to `int` via `strconv.Atoi` before storing in `DockerfileData`. The template literal `{{ .HostUID }}` in `ARG HOST_UID={{ .HostUID }}` also expects an int, not a string.
**Warning signs:** Template execution error about type mismatch, or `ARG HOST_UID=` line missing the value.

### Pitfall 5: Partial Git Config (Only Name or Only Email)

**What goes wrong:** Forwarding only `user.name` without `user.email` (or vice versa) produces a container where git commits fail with "Please tell me who you are."
**Why it happens:** Git requires both fields to commit.
**How to avoid:** Per locked decision — treat partial config as absent. Both must be non-empty to set `ForwardGitConfig = true`.
**Warning signs:** Entrypoint script has `git config --global user.name` but no matching `user.email` line.

### Pitfall 6: macOS Username With Special Characters Injected Into Dockerfile

**What goes wrong:** Dockerfile RUN command fails or creates wrong directory path.
**Why it happens:** macOS usernames can contain spaces on some systems. The Dockerfile interpolates `.MacOSUsername` directly into a `mkdir -p /Users/{{ .MacOSUsername }}` command.
**How to avoid:** Apply `[a-zA-Z0-9_.-]` regex sanitization (remove invalid chars) before storing in `DockerfileData.MacOSUsername`. Per locked decision, no additional length checks needed.
**Warning signs:** Dockerfile has `/Users/john doe` or similar in a RUN command.

### Pitfall 7: Non-Deterministic Container Names Across Machines

**What goes wrong:** Two engineers running zone on the same project get different container names, breaking shared tooling.
**Why it happens:** Using `filepath.Rel` or relative paths instead of `filepath.Abs` before hashing.
**How to avoid:** `filepath.Abs(repoPath)` MUST be called before SHA256. Spec §12 is explicit: "Deterministic from repo absolute path."
**Warning signs:** Different hashes for same repo on different machines, or tests with relative paths producing different names than expected.

### Generation Header Injection (Preserving BuildKit Syntax Directive)

```go
func injectGenerationComment(rendered, version string) string {
    header := "# Generated by zone " + version + " -- do not edit"
    lines := strings.SplitN(rendered, "\n", 2)
    if len(lines) >= 2 && strings.HasPrefix(lines[0], "# syntax=") {
        return lines[0] + "\n" + header + "\n" + lines[1]
    }
    // No syntax directive (entrypoint/shellrc) — prepend header
    return header + "\n" + rendered
}
```

## Open Questions

1. **Should `DockerfileData.HostUID` use `int` or `uint32`?**
   - What we know: `os/user.Current().Uid` is a string on all platforms; must be parsed. Docker uses positive int UIDs. Template uses `eq .HostUID 0` requiring numeric type.
   - What's unclear: Whether using `int` vs `uint32` matters for template comparison or Dockerfile ARG emission.
   - Recommendation: Use `int` — `strconv.Atoi` returns `int`, template `eq` comparison with literal `0` works, and UIDs fit in 32-bit positive int space.

2. **Where does `DockerfileData` get harness-specific fields (InstallCommands, HealthCheck, EntrypointCommand) in Phase 4?**
   - What we know: Phase 5 implements the harness interface that provides these values. Phase 4 stubs are empty.
   - What's unclear: Should Phase 4 export structs that compile but have zero-value harness fields, or define an interface that harnesses implement?
   - Recommendation: Define `DockerfileData` with all fields as concrete types (slices, strings). Phase 4 test fixtures populate them directly. Phase 5 wires harness to populate them. No interface needed in Phase 4.

3. **Should `RenderDockerfile` write to `.zone/Dockerfile` or just return a string?**
   - What we know: The cache dir writes established in Phase 3 use atomic writes via `cache.Cache`. Rendering and writing are separate concerns.
   - Recommendation: Render functions return strings. A separate `WriteRenderedFiles(c *cache.Cache, ...)` function calls `os.WriteFile` to `.zone/Dockerfile`, etc. This keeps rendering testable without filesystem.
